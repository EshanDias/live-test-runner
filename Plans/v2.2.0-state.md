# Live Test Runner — v2.2.0 State Reference

This document describes how the system works **as of v2.2.0**. It is a state reference, not an implementation plan.

---

## What v2.2.0 Added (full list)

- **Gutter status icons** — pass ✓ / fail ✗ / running ⟳ / pending ○ SVG icons in the editor gutter next to each `it()`/`test()` line
- **Inline duration text** — muted duration label rendered after the closing paren of each test block, colour-coded by threshold
- **CodeLens run button** — `▶ Run` above each `describe`, `it`, and `test` block, active only during a session
- **CodeLens debug button** — `▷ Debug` launches Jest via `vscode.debug.startDebugging` with `--runInBand --no-coverage`, scoped to that test via `--testNamePattern`
- **Jump to results** — `◈ Results` CodeLens on `it`/`test` lines with known results; focuses the Test Results panel and selects the matching row
- **Location data in result types** — `JestTestCaseResult` now carries `location?: { line, column }` from Jest's JSON output
- **Configurable duration thresholds** — `liveTestRunner.durationThresholds.*` settings with per-level (test/suite/file) amber and red values; propagated to all webviews via the `init` message
- **Shared duration utilities** — `src/utils/duration.ts` (extension-side) and `src/webview/utils.js` (webview-side) eliminate duplicated threshold logic
- **Suite rerun from editor** — clicking `▶ Run` on a `describe(...)` line now resolves the suite title via regex and reruns just that suite rather than the whole file

- Gutter column: `✓` green / `✗` red / `⟳` amber (spinning) / `○` grey

- `◈ Results` only appears on `it`/`test` lines (not `describe`) with a known result

## LineMap — Data Flow

The `LineMap` is a `Map<string, Map<number, LineEntry>>` stored inside `ResultStore`:

- **Key**: absolute file path → 1-based line number
- **Value**: `LineEntry` — `{ testId, suiteId, fileId, status, duration }`

### When it is populated

1. `run-started` (full suite) → `resultStore.clearAllLineMaps()`
2. `applyFileResultToStore()` (after each file result) → `resultStore.clearLineMap(filePath)`, then iterates `fileResult.testCases` and calls `setLineEntry()` for each test that has `location.line`
3. After populating: decorations are refreshed for any visible editor showing that file; `codeLensProvider.refresh()` fires

### When it is cleared

- Full run start: `clearAllLineMaps()`
- Per-file rerun: `clearLineMap(filePath)` before re-populating
- Session stop: `clearAllLineMaps()`

## Session Lifecycle — Decoration and CodeLens

| Event | Decorations | CodeLens |
|-------|-------------|----------|
| Session starts | Applied to all visible editors | `registerCodeLensProvider` called, lenses appear |
| File result arrives | `applyToEditor()` called for visible editors showing that file | `codeLensProvider.refresh()` fires |
| Editor becomes active | `decorationManager.applyToEditor()` called via `onDidChangeActiveTextEditor` | Lenses already registered |
| Session stops | `decorationManager.dispose()` clears and destroys all types | `codeLensDisposable?.dispose()` removes lenses |

## New Commands

| Command ID | Trigger | Behaviour |
|------------|---------|-----------|
| `liveTestRunner.rerunFromEditor` | `▶ Run` CodeLens click | If `it`/`test` line is in LineMap, reruns that test. If `describe` line, extracts the suite title via regex from source and reruns the suite. Falls back to the whole file if unresolved. |
| `liveTestRunner.debugFromEditor` | `▷ Debug` CodeLens click | Same scope resolution as rerun; launches `vscode.debug.startDebugging` with `--runInBand --no-coverage --testNamePattern` |
| `liveTestRunner.focusResult` | `◈ Results` CodeLens click | Focuses Test Results panel, calls `selectionState.select()` for that test |

## Duration Colour Thresholds

Thresholds are **per-level** and **user-configurable** via `liveTestRunner.durationThresholds.*` VS Code settings.

| Level | Green | Amber | Red |
|-------|-------|-------|-----|
| test  | < 100 ms | 100–500 ms | > 500 ms |
| suite | < 500 ms | 500–2000 ms | > 2000 ms |
| file  | < 1000 ms | 1000–5000 ms | > 5000 ms |

Threshold logic lives in `src/utils/duration.ts` (TypeScript, used by DecorationManager and webviews via init message) and is mirrored in `src/webview/utils.js` (plain JS globals for the webview sandbox). The extension reads current VS Code settings via `getThresholds()` and passes the result to each webview in the `init` message as `thresholds`; the webview calls `LiveTestUtils.setThresholds(msg.thresholds)` on receipt.

`describe` blocks do not show duration (Jest does not report suite-level duration in `location` entries).

## Source Restructure (also landed in v2.2.0)

The flat `src/` layout was replaced with a layered folder structure. No behaviour changed — pure organisation.

### New folder layout

```
src/
  extension.ts          ← activation and command wiring only (~110 lines)
  IResultObserver.ts    ← cross-cutting event contract, lives at root

  store/                ← pure data, no VS Code UI concerns
    ResultStore.ts
    SelectionState.ts

  session/
    SessionManager.ts   ← session lifecycle, run pool, on-save, rerun, debug

  framework/            ← everything that differs per test framework
    IFrameworkAdapter.ts
    JestAdapter.ts      ← extracted from extension.ts

  editor/
    CodeLensProvider.ts
    DecorationManager.ts

  views/
    BaseWebviewProvider.ts
    ExplorerView.ts     ← was TestExplorerProvider
    ResultsView.ts      ← was TestResultsProvider
```

### Key decisions

| Decision | Reason |
|---|---|
| `IResultObserver` at `src/` root | Cross-cutting contract used by every layer; not a store, not a type alias |
| `IFrameworkAdapter` inside `framework/` | Co-located with its implementations; adding Vitest = one new file |
| `SessionManager` extracted from `extension.ts` | `extension.ts` was doing three jobs — wiring, lifecycle, and run execution. Now it only wires |
| `types/` folder not created yet | No pure data-shape types exist yet; `RunScope`, `WorkspaceConfig` etc. will land here when needed |
| Webview assets stay in `src/webview/` | Read at runtime via `fs.readFileSync`, not compiled by tsc; `src/` signals "source files" correctly |

### What `extension.ts` now does (and doesn't do)

**Does:** create instances, register commands, subscribe to VS Code events, hand control to `SessionManager`.

**Doesn't:** know about `JestRunner`, `JestFileResult`, run concurrency, status bar text, or the on-save debounce.

## Known Limitations

- Jest only reports `location` when run with `--json` and the test uses a supported runner. Some Jest setups (CRA, Vitest) may not emit `location`, in which case no gutter icons appear for that file.
- The debug launch config uses a hardcoded `${workspaceFolder}/node_modules/.bin/jest` program path; projects with non-standard binary locations may need to adjust.
- Gutter icons are not animated at the VS Code API level — the spinning `⟳` SVG uses CSS `animateTransform` which VS Code renders statically in the gutter.
