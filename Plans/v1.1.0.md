# Live Test Runner — UI & Architecture Plan

## Background

The VS Code native Test Results panel has a Mac-specific bug where clicking an individual test
shows "The test case did not report any output." even when `run.appendOutput()` is called with the
test item parameter. This is a VS Code API limitation affecting Mac regardless of version.

The fix is to build a fully custom UI that we own entirely, giving us cross-platform control over
the test results experience — and the opportunity to build something closer to Wallaby.js in quality.

### Coverage gauge

- Horizontal progress bar, fills as tests complete during a run
- Shows percentage coverage number (e.g. `67%`)
- **Phase 1**: dummy / placeholder with a "feature coming soon" badge
- **Future**: real coverage from Jest `--coverage --coverageReporters=json`

### Explore Coverage button

- Disabled in Phase 1
- Shows a tooltip on hover: "Code coverage explorer — coming soon"
- **Future**: opens a coverage breakdown page inside the sidebar (file tree with % per file),
  with a back/home button to return to the main dashboard

### Run progress indicator

Shown only during an active run, above the summary table:

```
Running — 47 / 300 files complete
```

- Increments as each file result arrives (regardless of parallel order)
- Disappears when run finishes, replaced by the final summary
- Does not imply serial execution — just counts completions as they land
- No per-file "currently running" list needed; too noisy with parallel runs

### Summary table

Three columns always visible: `TOTAL` | `PASSED` | `FAILED`

Optionally a fourth `WAITING` column when a run is in progress (tests not yet executed).
Updates live during a run.

### Search / filter bar

- Text input, filters the Test List Layout in real time
- Matches against file name, suite name, or test case name
- Clears with an ✕ button
- Does not trigger any network call — pure client-side filter on the in-memory result tree

### Test List Layout (reusable component)

Used in both the Test Explorer sidebar and the Test Results panel column 1.

Three levels of expandable rows: **File → Suite → Test case**

Each row contains:

| Element | Detail |
|---------|--------|
| Expand toggle | `▶` / `▼` chevron, only on File and Suite rows |
| Status icon | `⟳` spinning (running) · `✓` green (passed) · `✗` red (failed) · `○` grey (not run) |
| Name | File path (shortened), suite name, or test case name |
| Duration badge | Colour-coded by threshold (see below). Has a tooltip. |
| Play button | `▶` icon button, reruns just this item. Visible on hover. |

**Duration colour thresholds (per test case):**

| Colour | Range | Tooltip text |
|--------|-------|-------------|
| 🟢 Green | < 100ms | Fast |
| 🟡 Amber | 100ms – 500ms | Could be improved |
| 🔴 Red | > 500ms | Slow — consider mocking heavy I/O |

For **test files**, thresholds are scaled: < 1s / 1s–5s / > 5s.
For **suites**, thresholds are: < 500ms / 500ms–2s / > 2s.

Thresholds will be user-configurable in settings in a future release.

**Behaviour:**
- Click any row → scopes the Test Results panel to that item (file / suite / test)
- Auto-expands files with failures when a run completes
- Passing files stay collapsed by default

## Panel — Output (Raw)

Standard VS Code `OutputChannel` registered as `Live Test Runner (ANSI)`.

- Every Jest command invoked is printed (with timestamp + working dir)
- Every line of stdout/stderr from Jest is streamed in real time
- No filtering, no parsing — raw terminal-style output
- ANSI colour codes are preserved (`createOutputChannel('...', 'ansi')`)
- No custom UI needed — VS Code's built-in Output panel handles this

## Panel — Test Results

A custom `WebviewViewProvider` registered as a panel-area view.

### Layout — three-column split

```
┌──────────────────┬─────────────────────────┬──────────────────────┐
│  Test List       │  Logs                   │  Errors              │
│  (same component │  [ Info ] [ Warnings ]  │                      │
│   as sidebar)    │  ─────────────────────  │  Stack traces        │
│                  │  console.log output     │  Assertion diffs     │
│  ← resizable →   │  custom output          │  Failure messages    │
│                  │                         │                      │
└──────────────────┴─────────────────────────┴──────────────────────┘
```

**Column 1 — Test list**
- Reuses the Test List Layout component
- Always visible — shows which item is selected and gives context for what logs + errors are showing
- User can resize columns by dragging the dividers
- Selection here scopes columns 2 and 3

**Column 2 — Logs** (two tabs)
- `Info` tab: `console.log` output, custom output, Jest's own stdout for the selected scope
- `Warnings` tab: `console.warn` output, deprecation notices, anything flagged as warning level
- ANSI colour rendering
- Auto-scrolls to bottom; locks if user scrolls up manually

**Column 3 — Errors**
- Stack traces from failed assertions
- Assertion diffs (expected vs received), formatted
- Jest failure messages
- Empty state: "No errors for this selection" when selected item passed

### Scoping rules (click to scope)

The selected item (from either the sidebar Test List or the panel Test List) drives what columns 2
and 3 show:

| Selection | Logs shows | Errors shows |
|-----------|-----------|-------------|
| File | All console output from all tests in file | All failures in file |
| Suite | All console output from tests in suite | All failures in suite |
| Test case | Console output for that test only | That test's failure (if any) |

### Selection sync

The selected item is held in **extension memory** as `selectedScope`. Both the Test Explorer
webview and the Test Results webview send a `select` message to the extension when the user clicks.
The extension updates `selectedScope` and broadcasts a `scope-changed` message to both webviews so
they stay in sync (the same row is highlighted in both lists).

## Architecture

### Component map

```
extension.ts (host)
  ├── JestRunner              — spawns Jest, streams output
  ├── ResultStore             — Map<id, TestResult>, holds all run data
  ├── SelectionState          — current selected scope (file/suite/test)
  ├── OutputChannel           — raw output panel (built-in VS Code API)
  ├── StatusBarItem           — Live Tests: ✅ / ❌ / ⟳
  ├── TestExplorerProvider    — WebviewViewProvider (sidebar)
  │     └── webview/explorer.html
  └── TestResultsProvider     — WebviewViewProvider (panel)
        └── webview/results.html
```

Both webviews share the same **Test List Layout** component — built as a single JS module
(`testListLayout.js`) that both HTML files import. This avoids duplicating logic.

### Message protocol

**Extension → both webviews**
```ts
{ type: 'run-started' }
{ type: 'run-finished',    passed, failed, skipped, duration }
{ type: 'file-started',    fileId, filePath, name }
{ type: 'file-result',     fileId, status, duration }
{ type: 'suite-started',   fileId, suiteId, name }
{ type: 'suite-result',    fileId, suiteId, status, duration }
{ type: 'test-started',    fileId, suiteId, testId, name }
{ type: 'test-output',     fileId, suiteId, testId, line, level: 'info'|'warn' }  // streaming
{ type: 'test-result',     fileId, suiteId, testId, status, duration, failureMessages }
{ type: 'scope-changed',   scope: 'file'|'suite'|'test', fileId, suiteId?, testId? }
{ type: 'coverage-update', percent }  // future
```

**Webview → extension**
```ts
{ type: 'select',   scope: 'file'|'suite'|'test', fileId, suiteId?, testId? }
{ type: 'rerun',    scope: 'file'|'suite'|'test', fileId, suiteId?, testId? }
{ type: 'search',   query: string }   // optional: could be client-side only
{ type: 'ready' }                     // webview signals it has initialised
```

### Parallelism and real-time output

Jest manages its own worker pool internally — the extension does not spawn or manage child workers
directly. Jest's default parallel mode runs test files across multiple worker threads, and your
extension simply streams whatever Jest writes to stdout/stderr.

Key points:

**Never use `--runInBand` for normal runs.** This flag forces Jest into single-threaded serial mode
and is only for debugging. Production runs always use Jest's default parallel workers.

**Output arrives interleaved and out of order.** With 300 files running in parallel, result messages
for different files will arrive in arbitrary order. The `ResultStore` handles this naturally since
every message is keyed by `fileId` — there is no assumption of ordering.

**The extension host must never block.** All Jest output is consumed on a `child_process` stream
(`stdout.on('data', ...)`). This is already non-blocking. The only risk is heavy JSON parsing on
large coverage outputs — that should be done with `JSON.parse` in a single call after the process
exits, not line-by-line during streaming.

**VS Code will not freeze** as long as the extension host thread is not blocked. Streaming stdout
line-by-line and posting messages to webviews is fast enough to be imperceptible. If this ever
becomes a concern in future (e.g. 1000+ file monorepos), the parsing can be offloaded to a
separate `Worker` thread using Node's `worker_threads` module — but this is not needed now.

**Current state:** Jest child process + stdout streaming is already implemented. This section
documents the model so it is not accidentally broken (e.g. by adding synchronous parsing mid-stream).

```ts
type TestResult = {
  id: string            // composite: `${fileId}::${suiteId}::${testId}`
  filePath: string
  suiteName: string
  testName: string
  status: 'running' | 'passed' | 'failed' | 'skipped' | 'pending'
  duration?: number
  outputLines: { text: string; level: 'info' | 'warn' }[]
  failureMessages: string[]
}

// Top-level store
resultStore: Map<fileId, {
  status, duration,
  suites: Map<suiteId, {
    status, duration,
    tests: Map<testId, TestResult>
  }>
}>
```

## Decisions Made

| Question | Decision |
|----------|----------|
| Single webview vs two separate | Two separate — sidebar + panel — sharing one JS component |
| Activity Bar or Testing sidebar | Activity Bar with custom test tube icon |
| Output format | Raw ANSI via built-in OutputChannel (no custom UI needed) |
| Auto-expand on failure | Yes — failed files expand automatically on run complete |
| Auto-scroll in logs | Scroll to bottom; locks if user has scrolled up |
| Columns 2+3: side-by-side vs tabs | Side-by-side (Logs \| Errors), with Info/Warnings tabs inside Logs |
| Column 1 in results panel | Always visible — user resizes columns by dragging dividers |
| Scoping logic lives in | Extension — data is pre-filtered then passed to webview on each selection |
| Selection sync mechanism | `SelectionState` in extension memory; selecting from either panel updates both |
| Coverage type | Deferred — decide when coverage phase is planned |
| Duration thresholds | Configurable in VS Code settings (future); sensible defaults baked in |
| Secondary sidebar | Not used |
