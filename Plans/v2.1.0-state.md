# Live Test Runner — v2.1.0 State Reference

This document describes how the system works **as of v2.1.0**. It is a state reference, not an implementation plan.

---

## What v2.1.0 Added

v2.1.0 upgraded columns 2 and 3 of the Test Results panel from a flat log dump to a **scoped, sectioned output display** with per-scope timestamps and structured error entries.

---

## Test Results Panel Layout

The panel has three resizable columns:

```
┌─────────────────┬──────────────────────────┬────────────────────────┐
│  Col 1: Tests   │  Col 2: Output           │  Col 3: Errors         │
│                 │  [All][Log][Info][Warn]   │                        │
│  File           │  [Error]                  │  ✗ test name           │
│    Suite        │                           │  <failure message>     │
│      ✓ test     │  📄 file.test.ts  just now│                        │
│      ✗ test     │  console.log output here  │  ✗ test name 2         │
│                 │                           │  <failure message>     │
│                 │  🔹 failing test  5s ago  │                        │
│                 │  console.log from test    │                        │
└─────────────────┴──────────────────────────┴────────────────────────┘
```

## Column 2 — Output (Scoped Sections)

### What is shown

Column 2 displays output sections based on the **currently selected row** in column 1.

| Selection | Sections shown |
|-----------|---------------|
| File row | File section (always). Suite sections only if that suite was individually rerun with output. Test sections only if that test was individually rerun with output. |
| Suite row | Suite section (always, even if empty — shows placeholder). Test sections within that suite only if individually rerun. |
| Test row | Test section (always, even if empty — shows placeholder). |

Each section has:
- A **header** with scope icon (📄 file / 🔷 suite / 🔹 test), label (file path or name), and a "captured N ago" timestamp.
- **Log lines** styled by level (log / info / warn / error).

### Where output comes from

Jest JSON reports console output at **file level only** via the `console` array. This means:

- **Full file run** → output stored at file scope. File section shows it.
- **Individual test rerun** (`--testNamePattern`) → output stored at both test and suite scope. Both sections show it when viewing that test or suite.
- **Suite-level rerun** → output stored at suite scope.

Suite and test sections are **never back-filled** from file-level output. If a test was never individually rerun, its section shows a placeholder.

### Tab filtering

The `[All] [Log] [Info] [Warn] [Error]` tabs filter all sections simultaneously. A section header is hidden when it has no visible lines under the active filter. Default tab is **All**.

## Column 3 — Errors

Column 3 shows structured failure entries scoped to the current selection.

| Selection | Errors shown |
|-----------|-------------|
| File row | All failed tests in the file, each with test name + failure messages. |
| Suite row | All failed tests in the suite. |
| Test row | That test's failure only, or a "No failures ✓" state. |

Each entry has the test name (in red) followed by the raw failure message in a monospace `pre` block. ANSI codes are stripped before storage.

## Data Flow

```
Jest JSON result arrives
  → applyFileResultToStore()
      → stores test statuses in ResultStore
      → calls setFileOutput / setSuiteOutput / setTestOutput
        depending on whether this was a full run or scoped rerun

User clicks a row in Col 1
  → webview posts  { type: 'select', scope, fileId, suiteId?, testId? }
  → TestResultsProvider handles 'select'
      → SelectionState.select()  (broadcasts scope-changed for list highlight)
      → _sendScopedLogs()        (builds and sends scope-logs)

File run completes, file matches current selection
  → extension.ts calls resultsProvider.sendScopedData()
      → _sendScopedLogs()        (columns 2 & 3 refresh live)
```

### Messages

| Direction | Type | Purpose |
|-----------|------|---------|
| Extension → both webviews | `scope-changed` | Highlight the selected row in column 1 |
| Extension → results only | `scope-logs` | Update column 2 (logSections) and column 3 (errorSections) |
| Extension → both webviews | `full-file-result` | Update column 1 test tree after a run |
| Extension → both webviews | `run-started` | Reset UI for a fresh full run |
| Extension → results only | `files-rerunning` | Mark specific files as running (partial rerun) |

#### `scope-logs` payload shape

```ts
{
  type: 'scope-logs',
  payload: {
    logSections: Array<{
      label:      string               // file path, suite name, or test name
      scope:      'file' | 'suite' | 'test'
      capturedAt: number | null        // Date.now() when output was stored; null = never run
      lines:      Array<{
        text:      string
        level:     'log' | 'info' | 'warn' | 'error'
        timestamp: number
      }>
    }>
    errorSections: Array<{
      label:  string
      scope:  'file' | 'suite' | 'test'
      errors: Array<{
        testName:        string
        failureMessages: string[]
        capturedAt:      number | null
      }>
    }>
  }
}
```

## ResultStore — Output Storage

`ResultStore` stores output per scope using `ScopedOutput`:

```ts
interface ScopedOutput {
  lines:      OutputLine[]   // the captured log lines
  capturedAt: number | null  // Date.now() when stored; null = never captured
}

interface OutputLine {
  text:      string
  level:     'log' | 'info' | 'warn' | 'error'
  timestamp: number
}
```

Each `FileResult`, `SuiteResult`, and `TestCaseResult` has an `output: ScopedOutput` field initialised to `{ lines: [], capturedAt: null }`.

Setters: `setFileOutput`, `setSuiteOutput`, `setTestOutput`  
Getters: `getFileOutput`, `getSuiteOutput`, `getTestOutput` (all return `EMPTY_OUTPUT` if not found — never `null`/`undefined`)

## What Does NOT Change in v2.1.0

- `testListLayout.js` — Column 1 rendering is untouched.
- `explorer.html` / `TestExplorerProvider.ts` — Sidebar is untouched.
- `packages/runner/` — No runner changes.
- Selection sync between the two webviews — unchanged.
- The `full-file-result` message still drives column 1 — only `outputLines` was removed from it.

## Known Limitations (to address in future)

- Console output is captured at file level by Jest JSON. There is no way to attribute a specific `console.log` call to an individual test without Jest instrumentation.
- Streaming output (`test-output` message type) is stubbed but not yet fully wired.
- `capturedAt` timestamps on individual `OutputLine`s are set to `Date.now()` at batch-store time, not the actual time the log was emitted.
