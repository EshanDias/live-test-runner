# Universal Jest Runner – Path B (Full Ownership Architecture)

## 🎯 Objective
Build a **framework-agnostic, deterministic Jest execution engine** that:
- Works across CRA, plain Jest, Next.js, Vite, etc.
- Always produces reliable JSON output
- Does NOT depend on framework CLIs at runtime
- Is future-proof and extensible

---

# 🧱 High-Level Architecture

```
Framework Detection → Binary Resolution → Config Resolution → Execution → Result Parsing
```

Each layer must be **independent, testable, and replaceable**.

---

# 1️⃣ Framework Detection

## Goal
Identify how tests are configured in the project.

## Detection Strategy

Inspect `package.json`:

### Rules
- If `react-scripts` in dependencies → `CRA`
- If `next` + `jest` config → `Next.js`
- If `vitest` present → `Vite`
- If `jest` config exists → `Jest`
- Else → `Unknown`

## Output

```
type Framework = 'cra' | 'jest' | 'next' | 'vite' | 'unknown'
```

---

# 2️⃣ Jest Binary Resolution

## Goal
Find the correct Jest executable reliably.

## Resolution Priority

1. `node_modules/.bin/jest`
2. Workspace root (monorepo support)
3. Direct path via package manager (pnpm/yarn)
4. Fallback (optional): `npx jest` (if allowed)

## Constraints
- Must NOT assume global install
- Must support pnpm symlinks
- Must work in CI

## Output

Absolute path to Jest binary

---

# 3️⃣ Config Resolution (Core Complexity)

## 3.1 Standard Jest Projects

### Sources
- `jest.config.js|ts|cjs|mjs`
- `package.json#jest`

### Strategy
- Load config directly
- Normalize into JSON

---

## 3.2 CRA Projects (Critical Path)

CRA hides Jest config internally.

### Primary Strategy (Recommended)

#### Step 1 – Locate react-scripts

Resolve path:

```
node_modules/react-scripts/bin/react-scripts.js
```

---

#### Step 2 – Extract Config

Run:

```
node react-scripts test -- --showConfig --passWithNoTests
```

Capture stdout → extract JSON → write to temp file

```
.cache/jest/cra-config.json
```

---

#### Step 3 – Cache Config

- Cache per project
- Invalidate if:
  - package.json changes
  - react-scripts version changes

---

### Fallback Strategies (in order)

#### Fallback 1 – Direct Node Execution

```
node <resolved-react-scripts-path> test -- --showConfig
```

#### Fallback 2 – Package Manager Execution

- `pnpm react-scripts`
- `yarn react-scripts`

#### Fallback 3 – npx (if allowed)

```
npx react-scripts test -- --showConfig
```

⚠️ Some environments disallow this → must be optional

---

### Hard Fallback (Last Resort)

Reconstruct minimal config:

- Use `babel-jest`
- Default `jsdom`
- Basic moduleNameMapper for:
  - CSS
  - images

⚠️ This will NOT be 100% accurate

---

# 4️⃣ Environment Normalization

## Required for CRA compatibility

Always enforce:

```
NODE_ENV=test
BABEL_ENV=test
CI=true
```

---

# 5️⃣ Execution Layer

## Command

```
jest --config <resolved-config> \
     --watchAll=false \
     --forceExit \
     --json \
     --outputFile=<tmpfile> \
     --no-bail
```

## Requirements

- MUST always disable watch mode
- MUST always produce JSON
- MUST never exit early (no-bail)

---

# 6️⃣ Result Handling

## Input

JSON output file

## Responsibilities

- Parse results
- Normalize structure
- Handle partial failures

## Edge Case

If file missing:
- Detect failure
- Retry OR fallback to raw stdout parsing

---

# 7️⃣ Caching Strategy

## Cache Items

- CRA extracted config
- Resolved binary path

## Cache Location

```
.cache/jest/
```

## Invalidation Rules

- package.json hash change
- lockfile change
- react-scripts version change

---

# 8️⃣ Monorepo Support

## Requirements

- Detect workspace root
- Resolve Jest per package
- Support mixed frameworks

---

# 9️⃣ Failure Handling Strategy

## Must Handle

### 1. Missing Jest
→ Clear error

### 2. CRA config extraction failure
→ Try fallbacks
→ Log reason

### 3. Invalid config
→ Validate JSON before execution

### 4. Test crashes
→ Still attempt to capture output

# 🧠 What You Might Be Missing (Important)

## 1. Process Control

Use spawn (NOT exec):
- stream stdout/stderr
- avoid buffer limits

---

## 2. Timeout Handling

- Kill long-running tests
- configurable timeout

---

## 3. Cross-Platform Paths

- Windows vs Unix `.bin`

---

## 4. Logging

- Debug logs for:
  - detected framework
  - resolved config
  - executed command

---

## 5. Security Constraints

- No `npx` in restricted environments
- No arbitrary command execution

---

## 6. Deterministic Runs

Ensure:
- Same inputs → same outputs
