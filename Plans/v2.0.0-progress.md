# v2.0.0 Implementation Progress

**Version decision:** v2.0.0 — replacing native VS Code Testing API with fully custom webview UI.  
**Plan source:** Plans/v1.1.0.md  
**Started:** 2026-04-03

---

## Why v2.0.0?
The plan replaces the entire `vscode.TestController` / `TestRun` API with custom `WebviewViewProvider` panels. This is a breaking architectural change — the native Test Results panel is gone, replaced by our own sidebar + panel views.

---

## Branch Strategy
Each feature area gets its own branch, merged to `develop`, then `main` on release.

  explorer.html, results.html, testListLayout.js, styles.css, updated extension.ts + package.json

### ✅ Bug-fix pass (develop 2026-04-03)

**Fixed:**
- **Explorer: tests only shown after run** — `runFiles()` now broadcasts `file-started` for each
  file before the run begins, so all files appear in `running` state immediately.
- **Column 1 empty when panel hidden** — Both `TestExplorerProvider` and `TestResultsProvider` now
  listen to `onDidChangeVisibility` and re-send `init` when the view becomes visible. VS Code drops
  `postMessage` calls to hidden views, so messages sent during a run were silently lost.
- **Column 2 always empty** — `JestRunner.parseJestJson` now extracts Jest's `console` array
  (present in `--json` output) into a new `consoleOutput` field on `JestFileResult`.
  `applyFileResultToStore` maps these to `OutputLine[]` (level: `log | info | warn`) and stores
  them at the file level via `resultStore.fileOutput()`. Console output from Jest JSON is captured
  at file granularity (not per-test), so it is displayed for file/suite scope selections.
- **Summary not updating** — follows from the Column 1 fix; `full-file-result` now reaches the
  Explorer even after panel visibility changes.
- **Test suites/tests never visible** — `testListLayout.js` now auto-expands all files and suites
  whenever data arrives (`setData` / `updateFile`), not only on failure.

**Output tab restructure (results.html Column 2):**
- 3 tabs: **Logs** (console.log), **Info** (console.info / error / debug / other), **Warn** (console.warn)
- Default active tab is **Logs**
- `full-file-result` now carries `outputLines` so Column 2 refreshes live when the selected file
  finishes without needing a re-selection.

**Data model changes:**
- `OutputLevel` type added: `'log' | 'info' | 'warn'` (was `'info' | 'warn'`)
- `FileResult.outputLines` added — file-level console output
- `ResultStore.fileOutput()` added — sets file-level output lines
- `JestConsoleEntry` interface added and exported from runner package

## Known Limitations (to address in future passes)

| Area | Issue |
|------|-------|
| Console output scoping | Jest JSON captures console at file level, not per-test. Selecting a suite or individual test shows the whole file's console output (best effort). Per-test scoping would require a custom Jest reporter. |
| Console in running state | Logs only appear after a file finishes (Jest JSON is produced at end of run). Real-time streaming would require a custom Jest reporter. |
| v1 bug — test suites in file | Fixed by auto-expand; if tests still don't appear check that `fileResult.testCases` is non-empty in the JSON output for your project. |

- Auto-expand all files/suites on data arrival (changed from: only failed files)
- Log auto-scroll locks if user has scrolled up
- Duration thresholds: test < 100ms green / 100-500ms amber / >500ms red
- Console output is file-scoped (Jest JSON limitation); per-test would need custom reporter
